# Checkpoint 01：需求分析参考输出

> 讲师兜底材料。现场应由 OpenCode 基于原始需求生成，不作为第一阶段输入。

## 1. 明确规则

| 规则编号 | 规则 | 来源 |
|---|---|---|
| R-01 | 只有系统管理员可以执行本需求中的用户维护操作 | 业务背景 |
| R-02 | 用户名必填，长度为 2～20 个字符 | 2.新增用户 |
| R-03 | 用户名必须唯一 | 2.新增用户 |
| R-04 | 用户昵称必填 | 2.新增用户 |
| R-05 | 密码必填 | 2.新增用户 |
| R-06 | 手机号选填；填写时格式必须合法 | 2.新增用户 |
| R-07 | 新增成功后可在用户列表中查询到 | 2.新增用户 |
| R-08 | 新增用户默认状态为正常 | 2.新增用户 |
| R-09 | 管理员可以删除普通系统用户 | 3.删除用户 |
| R-10 | 删除成功后列表不再显示该用户 | 3.删除用户 |
| R-11 | 当前登录用户不能删除自己 | 3.删除用户 |
| R-12 | 删除不存在用户不能影响其他用户 | 3.删除用户 |

## 2. 阻塞 AC 的问题

1. 用户名唯一性是否区分大小写？
2. “合法手机号”的可执行判断规则是什么？
3. 密码长度及复杂度是什么？
4. 删除成功的验证范围是否只有列表？
5. 失败场景要求什么样的状态和提示？

## 3. 非阻塞问题

1. 删除属于物理删除还是逻辑删除？验收可以先通过外部可见行为判断。
2. 新用户是否必须立即能够登录？原始需求没有要求。
3. 用户名允许哪些字符？需求只给出非空、长度、唯一性规则。
4. 手机号是否唯一？原始需求没有规定，不能自行形成 AC。

## 4. 风险

- 前端校验不代表 API 一定执行相同校验。
- “不能影响其他用户”需要设置对照用户验证。
- 测试数据若不唯一或不清理，会造成重复执行不稳定。
- 直接依据代码补充规则，会把当前实现误当成需求。
