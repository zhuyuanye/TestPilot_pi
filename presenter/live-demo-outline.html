Agent 应发现

  • 必填项与长度边界
  • 用户名大小写唯一性
  • 合法手机号定义
  • 删除后的列表/详情一致性

  • 自行补充密码复杂度
  • 把当前实现写成需求
  • 跳过阻塞问题直接生成 AC
  • 未经确认直接落盘
只有现场输入“AC 评审通过”，才写入 demo-live/outputs/。
测试设计 → 实现取证

分层验证，避免“UI 测一切”

API 自动化

必填、边界、唯一性、手机号、权限与删除一致性。

UI 自动化

只保留一个真实核心流程：登录、新增、搜索、删除。

人工测试

主观体验、探索性场景和不适合稳定自动化的检查。

实现取证：每项结论必须给出文件路径、类/方法和关键逻辑；静态分析只说明风险，不能宣告动态通过。
现场抽查至少两个源码位置，再输入“实现分析评审通过”。

反向验证

如何证明不是 LLM 编造？

让测试在正确环境下变绿，再由讲师故意破坏环境，亲眼看它变红。

RUOYI_UI_URL=http://localhost:9999 HEADLESS=true \
mvn -f demo-live/automation/ui/pom.xml test \
  2>&1 | tee demo-live/logs/ui-negative-challenge.log

challenge_status=${PIPESTATUS[0]}

① 非零退出码

不是只看 Agent 摘要。

② 新失败证据

失败截图、Trace、XML 同步产生。

③ 恢复后重跑

证明测试响应真实系统状态。

失败诊断

失败不是终点，错误分类才是价值

每个失败都回答

  • AC 预期是什么？
  • 真实响应或页面是什么？
  • 源码证据指向哪里？
  • 能否稳定复现并清理？

四类结论

  • 产品缺陷 实现不满足 AC
  • 测试脚本缺陷
  • 环境或配置问题
  • 未覆盖 / 未证实风险
底线：实现和需求冲突时，记录差异与缺陷；不改需求、不弱化断言、不把静态推测写成动态通过。
