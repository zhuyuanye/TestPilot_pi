# Prompt 02：现场生成并评审 AC

> 产品澄清由讲师在对话中提供；不要让 Agent 读取 `presenter/product-clarifications.md`，该文件仅是讲师兜底口径。

```text
请只使用：
- docs/requirements/user-management.md
- 本次对话中的需求分析
- 本次对话中产品刚刚给出的澄清

禁止读取仓库现有 outputs/、automation/ 和 presenter/checkpoints/。

先检查是否仍有阻塞验收的未决问题或冲突：
- 如果存在，停止生成并只列出阻塞项，等待人工决定；
- 如果不存在，生成候选验收标准。

AC 要求：
1. Given-When-Then 格式，编号 AC-01、AC-02……
2. 一条 AC 只表达一个主要业务规则
3. 覆盖正常、异常、边界、权限和删除后的数据一致性
4. 每条标注来源：需求章节或本次产品澄清编号
5. 不混入接口路径、数据库字段或当前代码行为
6. 不增加没有依据的规则
7. 附“需求规则—AC”追踪矩阵

生成候选稿后先在对话中做自检：
- 可追溯、可测试、无重复、无遗漏
- 没有主观模糊预期
- 没有把实现现状当需求

先展示候选稿和自检问题，不要立刻写文件。等待我人工评审；只有我回复“AC 评审通过”后，才写入 demo-live/outputs/user-management-acceptance-criteria.md，并停止。不要生成用例或代码。
```

## 人工控制点

至少抽查一条边界 AC、一条权限 AC 和一条删除一致性 AC。现场修正一次候选内容比直接展示完美结果更能体现协作过程。
