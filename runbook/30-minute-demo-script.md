## 屏幕操作

```bash

  demo-live/logs/ui-success-artifacts/

然后执行：

challenge_status=${PIPESTATUS[0]}
printf '%s\n' "$challenge_status" | tee demo-live/logs/ui-negative-challenge.exit-code
```

## 讲师说

> 现在测试必须变红，并且退出码必须是非零。如果它仍然显示通过，说明测试没有真正访问目标页面，或者执行证据链有问题。
>
> 大家现在看到，环境被故意破坏以后，测试确实失败了。失败日志和退出码也已经落地。
>
> 这一步说明测试结果会随着真实系统状态变化，而不是 LLM 预先编造的一段文字。

## 屏幕操作

恢复正确 URL，并说明正式结论使用恢复后最后一次正确环境执行结果。

可以不现场等待第二次完整运行，但必须说：

# 现场异常处理逐字稿

## 1. Agent 输出明显偏题

> 当前输出超出了本阶段范围。请停止生成代码，只回答本阶段要求的问题，并且不要读取根目录的兜底输出。

## 2. Agent 把实现现状当成需求

> 这条结论来自当前代码，不是原始需求或产品澄清。请把它移动到实现差异，而不是写入 AC。

## 3. Agent 为了通过准备修改断言

> 请停止修改断言。先展示对应 AC、测试输入和真实响应，判断这是产品缺陷、测试问题还是环境问题。没有新的需求依据，不允许降低断言。

## 4. Agent 生成超过时间预算

> 当前阶段已经超过现场时间预算。为了保证完整流程，我现在只启用这一阶段的兜底，并且会留下明确记录。后续测试仍然现场重新执行。

## 5. 后端或 Vue 临时不可用

> 当前失败发生在环境可用性检查阶段，还没有进入业务断言，因此不能归类为产品功能缺陷。我会展示环境诊断，并切换到经过验证的执行证据兜底。

## 6. UI 有头执行受窗口或动画影响

> 当前问题属于 UI 执行稳定性，不代表业务结论。我先切换无头模式确认脚本，再决定是否恢复有头展示；API 业务规则结果不受这个窗口问题影响。

## 7. 必须使用兜底时

> 我现在明确启用【当前阶段】的预置兜底。系统会在 `demo-live/FALLBACK_USED.md` 中记录时间、阶段和来源提交。兜底只替代当前受阻阶段，不替代前面已经现场完成的过程。
