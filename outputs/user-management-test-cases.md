# 用户管理测试用例

## 1. 测试范围

- 测试依据：`outputs/user-management-acceptance-criteria.md` 中的 AC-01～AC-25。
- 业务规则优先通过 API 自动化验证。
- UI 自动化仅保留一条用户新增、查询和删除的核心流程。
- 删除结果只通过用户列表和用户详情查询验证，不直接检查数据库物理记录。
- 不验证新增用户能否立即登录。
- 错误信息不逐字匹配，但必须能够明确说明对应的失败原因。

## 2. 优先级定义

| 优先级 | 判定标准 |
|---|---|
| P0 | 核心新增或删除流程、关键必填规则、唯一性、权限控制或可能影响其他用户的数据安全规则；失败会阻塞主要业务或造成高影响。 |
| P1 | 重要格式、边界和数据一致性规则；失败不会完全阻塞主流程，但会导致错误数据或明显业务偏差。 |
| P2 | 已明确允许的宽松规则或较低风险兼容性场景；适合在核心规则稳定后执行。 |

## 3. 测试数据与清理策略

| 项目 | 约定 |
|---|---|
| 运行标识 | 每次执行生成恰好 6 位小写字母或数字组成的 `{R6}`。同一次执行中保持不变，不同执行使用不同值。 |
| 常规用户名 | 使用 `u{R6}tNN`，其中 `NN` 为用例编号；总长度不超过 20 个字符。 |
| 昵称 | 使用 `N-{R6}-TCNN`，用于识别本次执行创建的数据。 |
| 默认有效密码 | 使用 `abcde`，长度为 5 个字符。 |
| 默认有效手机号 | 使用 `13800000000`。手机号没有唯一性要求，可以重复使用。 |
| 前置清理 | 执行前通过用户查询确认本用例用户名不存在；如果存在同名测试数据，通过系统删除能力清理后再执行。不得删除非测试前缀用户。 |
| 后置清理 | 每个成功新增且未在用例中删除的用户，都通过系统删除能力清理；随后确认用户列表和用户详情均查询不到该用户。 |
| 失败场景 | 验证新增未成功，不应产生待清理用户；如发现残留，记录缺陷并使用系统删除能力清理。 |
| 最小长度用户名 | 长度为 2 的用户名使用测试环境保留值 `q2`。执行前后均清理该用户，避免重复执行冲突。 |
| 管理员夹具 | 使用预置且可恢复的测试管理员执行操作；除自删除验证外，不将其作为任何删除目标。 |

| 步骤 | 操作 | 预期结果 |
|---|---|---|
| 1 | 进入系统管理中的用户管理功能并发起新增用户。 | 新增用户操作可用。 |
