# 第四步：需求与实现对照

## 进入条件

- 已加载 `test-workflow-core`。
- 需求、AC、测试用例均已人工批准。
- 已明确目标源码路径。

## 执行步骤

1. 定位 UI 表单规则、请求封装和核心页面。
2. 定位 Controller、Service、领域对象和参数校验。
3. 定位认证、创建、查询、详情、更新和删除能力。
4. 检查唯一性、大小写、长度、格式、权限和状态处理。
5. 识别测试数据准备与清理接口。
6. 检查现有测试框架和可复用工具。
7. 每项结论分成：
   - AC 预期
   - 实现证据
   - 差异、风险或待动态验证项
8. 实现证据必须给出文件路径和类、方法、组件或函数。

## 禁止行为

- 只凭经验猜测实现。
- 因为静态代码看似正确就标记测试通过。
- 在黑盒测试阶段直接修改业务源码。
- 把现有实现反写成 AC。

## 候选输出

写入 `drafts/04-implementation-analysis.md`，包含实现证据矩阵、潜在差异、动态验证建议和测试工程约束。

## 完成门禁

设置 `draft_ready` 并停止。人工必须打开并核对代表性源码位置后才可批准。
