# 第三步：测试设计与分层

## 进入条件

- 已加载 `test-workflow-core`。
- AC 已人工批准。

## 约束

此阶段只根据需求和 AC 设计，不读取实现代码，不因为系统当前能力降低覆盖。

## 执行步骤

1. 每条用例分配唯一 TC 编号并关联 AC。
2. 写明前置条件、动态数据、步骤、客观预期和清理方式。
3. 使用等价类、边界值、错误推测和状态迁移。
4. 标注 P0/P1/P2及理由。
5. 标注 API 自动化、UI 自动化或人工测试层。
6. API 优先验证业务规则、边界和权限。
7. UI 默认只保留少量核心用户旅程，避免重复所有 API 边界。
8. 测试数据必须唯一、可重复、可恢复，失败后也能清理。
9. 输出 AC—TC—执行层矩阵并列出未覆盖 AC。
10. 高风险场景必须声明安全开关和恢复方案。

## 候选输出

写入 `drafts/03-test-cases.md`，包含：

- 用例明细
- 优先级和设计方法
- 自动化分层理由
- 数据与清理策略
- AC—TC 追踪矩阵
- 未覆盖项和风险

## 完成门禁

设置 `draft_ready` 并停止。人工重点评审边界、权限、状态一致性、分层和数据清理后才可批准。
