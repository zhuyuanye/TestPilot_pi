# 第五步：API 自动化

## 进入条件

- 已加载 `test-workflow-core` 并阅读证据规范。
- 实现分析已人工批准。
- 已明确测试环境、技术栈和凭据环境变量名称。

## 第一门禁：计划

首次执行只能输出计划，包含：

- TC/AC 范围和优先级
- 工程位置和拟创建文件
- 认证方式
- HTTP、业务状态和关键字段断言
- 动态数据生成
- Before/After/finally 清理
- 日志、退出码和机器可读结果
- 高风险测试开关
- 预计执行命令和失败风险

设置 `plan_ready` 并停止。不得在计划批准前编码。

## 计划批准后

1. 在隔离测试工程中实现，不修改业务源码。
2. URL 和凭据只从环境变量读取。
3. 测试互相独立，不依赖执行顺序。
4. 同时断言 HTTP 状态、业务状态和关键字段。
5. 每个测试关联 TC/AC。
6. 动态生成唯一数据，失败后也尽力清理。
7. 不打印密码、完整 Token 和授权头。
8. 先编译，再运行 P0，再运行边界或全量范围。
9. 使用 `tee` 保存完整日志，并保存原始退出码。
10. 保留 Surefire XML/JSON 等机器可读结果。

## 失败诊断

先展示 AC、输入和实际响应，再分类。不得直接修改断言。清理失败必须作为风险报告。

## 结果候选

写入 `drafts/05-api-result.md`，并将状态设为 `result_ready`。内容必须由当前运行原始证据生成。

## 第二门禁：结果

人工检查代码、日志、退出码、机器结果、失败分类和清理后才可批准进入 UI 阶段。
