# 第八步：最终评审与人工收口

## 原则

本阶段只读，不修改测试、断言、业务代码和原始证据。

## 审查内容

1. 验证双向追踪：

```text
需求 ↔ 产品澄清 ↔ AC ↔ TC ↔ 自动化方法 ↔ 原始结果 ↔ 缺陷
```

2. 明确区分：
   - 动态验证通过
   - 动态验证失败
   - 仅静态分析
   - 未覆盖
   - 环境阻塞
   - 已跳过
   - 使用兜底
   - 原始证据不足、未证实
3. 检查是否通过弱化断言隐藏失败。
4. 检查日志、退出码、机器结果、Trace、截图和哈希。
5. 检查清理失败和残留数据风险。
6. 检查密码、Token、授权头和敏感 Trace 泄露。
7. 列出未覆盖 AC、剩余风险和后续建议。
8. 基于证据建议：`通过`、`有条件通过` 或 `不通过`。

## 输出

生成最终评审候选，包含：

- 结论建议及依据
- 追踪完整性
- 动态/静态/未证实分类
- 缺陷和未覆盖风险
- 兜底使用情况
- 产品、开发、测试空白签字栏

## 边界

Agent 的结论只是建议。不得自行签字、批准需求验收或声称团队已经接受风险。
